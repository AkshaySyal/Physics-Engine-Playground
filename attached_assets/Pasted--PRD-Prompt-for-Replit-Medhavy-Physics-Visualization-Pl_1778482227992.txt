

PRD Prompt for Replit: Medhavy Physics Visualization Playground
Build a modern web application called Medhavy Physics Visualization Playground.
The purpose of this application is to help us determine mappings between:
physics concepts / user queries → appropriate visualization types → best rendering framework
This app will be used by professors and researchers to compare different visualization frameworks for physics education.
The system should not only generate graph plots. It should support:
plots
vector diagrams
free-body diagrams
conceptual illustrations
animations
textbook-style physics visuals
This is a research/demo platform for the Medhavy AI / OpenStax University Physics Volume 1 project.

1. Core Product Goal
Given a physics user query, the app should:
classify the query into a physics visualization category
determine which visualization frameworks are appropriate
generate only plot/diagram/visual outputs, not long textual answers
render outputs side-by-side across selected frameworks
allow professors to compare which framework creates the clearest educational visual
The app should focus on visualization quality, not final answer explanation.

5. Frameworks to Support
The app must support the following 5 visualization/rendering frameworks:
Matplotlib
Recharts
D3.js
Manim
Google image-generation model, such as Gemini Flash image generation or equivalent
Use judgment: not every framework should generate every visual type.
For example:
Matplotlib is good for plots, not polished free-body diagrams.
Recharts is good for interactive charts, not free-body diagrams.
D3.js is good for SVG-based diagrams and browser-native visuals.
Manim is good for polished animations.
Google image generation is good for textbook-like diagrams, free-body diagrams, and conceptual illustrations.

6. Framework Selection UI
The UI must allow the user to select or deselect visualization frameworks.
By default, all frameworks should be selected.
The professor should be able to compare:
all 5 frameworks
only 2 selected frameworks
only one framework
For each submitted query, the app should generate only outputs for selected frameworks that are appropriate for the detected visualization category.
If a selected framework is unsuitable, show a clear placeholder card:
“Not suitable for this visualization type.”

7. Output Requirements
For every submitted query, produce visual outputs only.
Do not generate long text explanations.
Each output card should contain:
framework name
visualization type
rendered visual
generation status
short one-line note if needed
rendering time if available
The comparison layout should be side-by-side.
Example:
Query: Compare projectile trajectories for 30°, 45°, and 60°

[Matplotlib Output] [Recharts Output] [D3 Output]

[Manim Output] [Google Image Output / Not Suitable]


8. Query Generation Feature
Each category card should have:
hardcoded example questions
“Generate Similar Question” button
When clicked, the app should use the LLM to generate a new question similar to that category.
Requirements:
stay within same category
use realistic physics language
include parameters when useful
produce questions suitable for visualization
Example:
Category: Projectile Motion
Hardcoded question:
“Compare projectile trajectories for 30°, 45°, and 60°.”
Generated similar question:
“Plot projectile trajectories for launch speeds of 10 m/s, 20 m/s, and 30 m/s at a fixed angle of 45°.”

9. Visual Generation Pipeline
When a query is submitted:
Step 1: Classify Intent
Classify into schema:
{
  "query_intent": "force_balance",
  "physics_concept": "pulley tension",
  "visual_type": "free_body_diagram",
  "recommended_frameworks": ["d3", "google_image_generation"],
  "requires_animation": false,
  "requires_numeric_data": false
}

Step 2: Select Frameworks
Intersect:
frameworks selected by user
frameworks appropriate for detected visualization type
Step 3: Generate Visuals
Generate the output independently for each framework.
Step 4: Render Comparison
Render outputs in comparison grid.

10. Framework-Specific Implementation Guidance
Matplotlib
Use for:
kinematics plots
trajectory plots
wave plots
oscillation plots
energy curves
Return:
image file
base64 PNG
SVG
Avoid for:
complex free-body diagrams unless simple

Recharts
Use for:
browser-native interactive plots
line charts
bar charts
parameter comparisons
energy bar charts
Return:
structured JSON data
chart configuration

D3.js
Use for:
vector diagrams
free-body diagrams
custom SVG diagrams
annotated force diagrams
conceptual diagrams
Return:
SVG elements or structured JSON for D3 rendering

Manim
Use for:
animations
wave propagation
circular motion
oscillations
projectile motion animation
force decomposition animation
Return:
MP4 or GIF if rendering is available
otherwise generated Manim script preview
mark status as async if rendering takes time

Google Image Generation Model
Use for:
textbook-style diagrams
free-body diagrams
conceptual illustrations
annotated visual explanations
Return:
generated image
image URL or base64
Prompt requirements:
clean textbook style
white or dark scientific background
labeled force arrows
correct physics notation
minimal clutter
educational clarity

13. Important Research Framing
This application exists to answer:
Which visualization framework produces the most useful educational visual for a given physics query type?
Therefore, prioritize:
comparison
controllable framework selection
query-category taxonomy
visual clarity
easy demo flow

14. Expected Deliverable
Build a working deployed web app that lets professors:
choose a physics query category
select a hardcoded example or generate a similar question
select visualization frameworks
submit the query
compare visual outputs side-by-side
The output should make it easy to decide whether Matplotlib, Recharts, D3.js, Manim, or Google image generation is best for that physics visualization category.
